# Vérifications d'accès par route

Ce document décrit, pour chaque route protégée par une condition d'accès, où (couche/middleware) et comment la vérification est effectuée dans le code.

| Endpoint | Condition d'accès | Couche de vérification | Ce qu'on vérifie concrètement |
|---|---|---|---|
| POST /auth/login | N/A (authentification) | Route `src/routes/auth.js` | Vérifie `email`/`password` en base, journalise succès/échec, génère tokens JWT via `src/auth.js`. |
| POST /auth/refresh | N/A (authentification) | Route `src/routes/auth.js` + `src/auth.js` | Vérifie `refreshToken` via `verifyRefreshToken`, retrouve l'utilisateur, renvoie nouveaux tokens. |
| GET /sensors/:sensorId | role ∈ {admin, operator, reader, device} ; opérateur => seulement si capteur dans sa zone | Middleware `authenticate` (src/auth.js), `requireScope("sensors:read")`, `requireRoles(...)`, `requireSensorAccess` (src/authorization.js) | `requireSensorAccess` : recherche le capteur, si token device => vérifie ownership (`resourceType==='sensor'` et `resourceId`), sinon admin passe, sinon compare `user.zone === sensor.areaId`. En cas d'échec retourne 404 ou 401/403 selon le cas et journalise. |
| POST /sensors | role ∈ {admin, operator} ; opérateur => only pour sa zone | `authenticate`, `requireScope("sensors:write")`, `requireRoles("admin","operator")` (middleware) + validation inline (src/routes/sensors.js) | Le handler vérifie l'existence de `areaId` et, pour `operator`, compare `req.user.zone === areaId`. Si zone non autorisée retourne 404 (message spécifique). |
| GET /areas | role ∈ {admin, operator, reader} | `authenticate`, `requireScope("areas:read")`, `requireRoles(...)` + `filterAreasForUser` (src/authorization.js) | `filterAreasForUser` : admin → toutes les zones ; operator/reader → retourne uniquement la zone `user.zone`; device → aucune zone. |
| POST /areas | role === admin | `authenticate`, `requireScope("areas:write")`, `requireRoles("admin")` | Middleware garantit role/admin ; le handler valide payload (buildingId, name) et crée la zone. |
| GET /areas/:areaId/actuators | role ∈ {admin, operator, reader} ; opérateur/reader seulement si zone | `authenticate`, `requireScope("actuators:read")`, `requireRoles(...)`, `requireAreaAccess` (src/authorization.js) | `requireAreaAccess` : recherche la zone, puis `canAccessArea(user, area)` — admin passe ; operator/reader doivent avoir `user.zone === area.id`. Sinon 404 et journalisation. |
| POST /areas/:areaId/actuators | role ∈ {admin, operator} ; opérateur seulement si zone | `authenticate`, `requireScope("actuators:write")`, `requireRoles(...)`, `requireAreaAccess` | Comme ci-dessus ; le handler valide `type`/`state` et crée l'actionneur. |
| GET /actuators/:actuatorId | role ∈ {admin, operator, reader, device} ; opérateur/reader => zone | `authenticate`, `requireScope("actuators:read")`, `requireRoles(...)`, `requireActuatorAccess` (src/authorization.js) | `requireActuatorAccess` : recherche actionneur ; si token device → vérifie ownership (`resourceType==='actuator'` et `resourceId`), sinon admin passe, sinon compare `user.zone === actuator.areaId`. En cas d'échec 404 et log. |
| PUT /actuators/:actuatorId | role ∈ {admin, operator} ; opérateur seulement si zone | `authenticate`, `requireScope("actuators:write")`, `requireRoles(...)`, `requireActuatorAccess` | Même logique que GET, puis le handler valide et modifie `state`. |
| DELETE /actuators/:actuatorId | role === admin | `authenticate`, `requireScope("actuators:write")`, `requireRoles("admin")`, `requireActuatorAccess` | `requireActuatorAccess` vérifie existence + accès ; la suppression n'est autorisée qu'aux admins (middleware). |
| GET /areas/:areaId/alert-thresholds | role ∈ {admin, operator, reader} ; opérateur/reader => zone | `authenticate`, `requireScope("alert-thresholds:read")`, `requireRoles(...)`, `requireAreaAccess` | `requireAreaAccess` garantit que l'utilisateur peut accéder à la zone, puis le handler filtre les seuils pour les capteurs de la zone. |
| POST /areas/:areaId/alert-thresholds | role ∈ {admin, operator} ; opérateur seulement si zone ; capteur doit appartenir à la zone | `authenticate`, `requireScope("alert-thresholds:write")`, `requireRoles(...)`, `requireAreaAccess` + validation inline (src/routes/alertThresholds.js) | `requireAreaAccess` vérifie accès à la zone ; le handler vérifie que `sensorId` appartient bien à la `areaId` (sinon 400). |
| GET /sensors/:sensorId/measures | role ∈ {admin, operator, reader, device} ; opérateur/reader => zone | `authenticate`, `requireScope("measures:read")`, `requireRoles(...)`, `requireSensorAccess` | `requireSensorAccess` : voir vérification capteur (ownership device ou `user.zone === sensor.areaId`), puis le handler retourne les mesures. |
| POST /sensors/:sensorId/measures | role ∈ {admin, device} ; device → ownership ; capteur doit être actif | `authenticate`, `requireScope("measures:write")`, `requireRoles("admin","device")`, `requireSensorAccess` | `requireSensorAccess` vérifie accès ; le handler vérifie `sensor.status !== 'inactive'` (sinon 409) et valide payload (timestamp, value). |
| POST /users | role === admin | `authenticate`, `requireScope("users:write")`, `requireRoles("admin")` | Middleware empêche les non-admins ; le handler vérifie payload et crée l'utilisateur. |
| GET /users/:userId | role ∈ {admin, operator, reader} ; sinon accès à son propre user | `authenticate`, `requireScope("users:read")`, `requireRoles(...)`, `requireUserAccess` (src/authorization.js) | `requireUserAccess` : recherche l'utilisateur cible ; admin passe ; sinon seul `targetUser.id === user.id` est autorisé. Sinon 404 et log. |

## Notes générales

- Les vérifications d'authentification JWT sont centralisées dans `src/auth.js` (`authenticate`).
- Les vérifications d'autorisation (scopes, rôles, accès BOLA, ownership device) sont dans `src/authorization.js` (`requireScope`, `requireRoles`, `requireAreaAccess`, `requireSensorAccess`, `requireActuatorAccess`, `requireUserAccess`).
- Certaines validations spécifiques (p.ex. « le capteur appartient à la zone » ou « operator only for his zone » lors de créations) sont implémentées inline dans les handlers de route (ex. `POST /sensors`, `POST /areas/:areaId/alert-thresholds`). Ces vérifications doivent rester dans le handler lorsqu'elles réfèrent au body (p.ex. `areaId`) mais peuvent réutiliser les middlewares d'autorisation pour la partie identité/zone.
